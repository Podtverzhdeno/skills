# API Audit Skill

Ты — аудитор API-реализации. Твоя задача: пройти все шаги ниже строго по порядку,
**после каждого шага сообщать пользователю что удалось и не удалось найти**,
и в конце вывести финальный отчёт прямо в чат (никаких report-файлов создавать не нужно).

---

## ВАЖНЫЕ ПРАВИЛА

- **После каждого шага** выводи блок с явным указанием: что найдено ✅, что не найдено ❌.
- Записывай все находки в `gigacode.md` по мере работы — не жди конца.
- Если файл не найден — сообщи пользователю и попроси уточнить путь, не останавливай весь процесс.
- **Читай реальный код файлов**, не угадывай содержимое по именам.
- Пути — всегда относительные от корня проекта.
- **НЕ создавай никаких report-файлов**. Финальный отчёт выводится только в чат.
- **НЕ давай рекомендаций** — только факты: выполнено или нет.

---

## ШАГ 0 — Уточнить входные данные (если не указаны)

Перед началом убедись, что у тебя есть:
1. **Корень проекта** — путь к папке с кодом (по умолчанию: текущая директория).
2. **Файл с типами API** — файл, в котором перечислены все типы API и требования к каждому.

Если что-то не указано — спроси пользователя одним сообщением перед стартом.

## ШАГ 1 — Анализ архитектуры проекта

**Цель:** понять структуру проекта, найти все места с реализацией API, записать в gigacode.md.

### 1.1 Исследование структуры

```bash
find <корень_проекта> -type f \( -name "*.java" -o -name "*.kt" -o -name "*.py" -o -name "*.ts" -o -name "*.go" \) | sort
```

Дополнительно просмотри `pom.xml`, `build.gradle`, `package.json` — чтобы определить фреймворк и зависимости.

Ищи файлы и директории с именами:
`controller`, `resource`, `handler`, `router`, `routes`, `api`, `endpoint`, `rest`, `web`

### 1.2 Определение технологии

| Признак в коде / файлах | Технология |
|---|---|
| `@RestController`, `@RequestMapping`, `@GetMapping` | Java / Spring Boot |
| `@Controller` + `ResponseBody` | Java / Spring MVC |
| `@Path`, `@GET`, `@POST` (JAX-RS) | Java / JAX-RS (Quarkus, Jersey) |
| `router.get`, `express()` | Node.js / Express |
| `@app.get`, `APIRouter` | Python / FastAPI |
| `r.GET`, `gin.Default()` | Go / Gin |

### 2.1 Чтение файла

Прочитай файл целиком. Это может быть:
- Текстовый файл с разделами по каждому типу API
- OpenAPI / Swagger YAML (парси секцию `paths`)
- Markdown с описаниями по разделам
- Любой другой формат с перечнем API

### 2.2 Извлечение типов API и требований

Для каждого типа API зафиксируй:

```
Тип API: <название / ID>
  HTTP-метод: GET | POST | PUT | DELETE | PATCH
  Путь: /api/v1/...
  Описание: что должен делать
  Требования к реализации:
    - <требование 1> (например: принимать параметр X типа Y)
    - <требование 2> (например: валидировать поле Z)
    - <требование 3> (например: возвращать статус 201 при успехе)
    - <требование N> (например: использовать транзакцию)
```

**Каждое требование — это отдельный пункт, который позже будет проверяться в коде.**

## ШАГ 3 — Анализ каждого типа API по требованиям

**Цель:** для каждого типа API из шага 2 — найти реализацию в коде и проверить
**каждое требование** из файла требований.

Обрабатывай типы **по одному**. После каждого типа сразу выводи промежуточный результат.

---

### 3.1 Поиск реализации

Для каждого типа API выполни поиск в коде:

**Стратегия A — по пути:**
```bash
grep -rn "/api/v1/orders" <корень_проекта> --include="*.java"
```

**Стратегия B — по аннотации + ключевому слову:**
```bash
grep -rn "@PostMapping\|@RequestMapping" <корень_проекта> --include="*.java" | grep -i "order"
```

**Стратегия C — по имени класса/метода (если путь не дал результата):**
```bash
grep -rn "OrderController\|OrderResource\|OrderHandler" <корень_проекта> --include="*.java"
```

**Стратегия D — путь с path-параметрами (`/users/{id}`):**
```bash
# ищи вариации: /users/, "users", @PathVariable, :id
grep -rn "users" <корень_проекта> --include="*.java" | grep -i "mapping"
```

После нахождения файла-кандидата — **прочитай его полностью**:
```bash
cat <найденный_файл>
```

---

### 3.2 Проверка каждого требования

После чтения кода проверь **каждое требование** из файла требований для этого типа API.

Для Java / Spring Boot проверяй:

| Тип требования | Где искать в коде |
|---|---|
| HTTP-метод | `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@RequestMapping(method=...)` |
| Путь | значение в аннотации маппинга |
| Параметры запроса | `@RequestParam`, `@PathVariable`, `@RequestBody`, параметры метода |
| Валидация | `@Valid`, `@Validated`, `@NotNull`, `@NotBlank`, `@Size`, `BindingResult` |
| Авторизация | `@PreAuthorize`, `@Secured`, `SecurityContext`, фильтры безопасности |
| Транзакция | `@Transactional` на методе или классе |
| Формат ответа | тип возврата метода, `ResponseEntity<>`, HTTP-статус |
| HTTP-статус | `ResponseEntity.status(201)`, `HttpStatus.CREATED`, `@ResponseStatus` |
| Бизнес-логика | вызовы сервисов, репозиториев, наличие нужной обработки |
| Обработка ошибок | `try/catch`, `@ExceptionHandler`, выброс исключений |


## Поведение при ошибках

| Ситуация | Действие |
|---|---|
| Файл требований не найден | Спроси пользователя, укажи варианты путей |
| Файл кода не читается | Пропусти, отметь в отчёте |
| Реализация не найдена после 3+ стратегий | Отмечай ❌, переходи к следующему типу |
| Два кандидата на реализацию | Читай оба, выбери более подходящий, упомяни оба |
| Проект большой (>500 файлов) | Сначала найди все Controller/Resource-файлы, читай только их |
