# Skill

## Назначение

Этот Skill предназначен для слабой или средней модели, которая должна:

- полностью проанализировать архитектуру проекта;
- найти файл с требованиями к API;
- сопоставить требования с реализацией в коде;
- определить, выполнены требования или нет;
- составить подробный отчет;
- постоянно показывать пользователю прогресс;
- сохранять результаты анализа в `gigacode.md`.

Skill должен заставлять модель действовать строго по шагам.

---

# Главная задача

Нужно:

1. Изучить проект полностью.
2. Найти все типы API.
3. Найти файл с требованиями.
4. Разбить требования по типам API.
5. Найти реализацию каждого API в проекте.
6. Проверить соответствие требованиям.
7. Составить итоговый отчет.
8. Записывать все результаты в `gigacode.md`.
9. После каждого действия показывать пользователю:
   - что найдено;
   - что не найдено;
   - что удалось подтвердить;
   - что не удалось проверить.

---

# Очень важные правила

## Запрещено

- Делать выводы без проверки кода.
- Придумывать реализацию, если она не найдена.
- Считать требование выполненным без подтверждения.
- Пропускать шаги.
- Игнорировать отсутствующие файлы.

---

## Обязательно

- Всегда указывать пути к файлам.
- Всегда объяснять вывод.
- Всегда разделять:
  - найдено;
  - не найдено;
  - найдено частично;
  - не удалось проверить.
- После каждого этапа обновлять `gigacode.md`.
- Писать коротко и понятно.
- Делать один логический шаг за раз.

---

# ШАГ 2. Поиск файла требований к API

Нужно найти документ, где описано:

- какие API должны существовать;
- какие эндпоинты должны быть;
- какие поля обязательны;
- какие ограничения существуют;
- какие правила должны соблюдаться.

Искать:

- markdown-файлы;
- docs;
- OpenAPI;
- swagger;
- yaml/json спецификации;
- excel/csv/pdf/txt;
- README;
- Confluence-export;
- любые документы с требованиями.

## Если найдено несколько документов

Нужно:

- перечислить все;

## Если ничего не найдено

Нужно:

- явно написать это;
- продолжить анализ по коду;
- отметить риск.

## Что записать в gigacode.md

- найденные документы;

# ШАГ 3. Разделение требований по типам API

Нужно прочитать требования и разбить их на группы.

Для каждого типа API определить:

- название API;
- назначение;
- обязательные требования;
- необязательные требования;
- ограничения;
- ожидаемые входные данные;
- ожидаемые выходные данные;
- требования к безопасности;
- требования к валидации;
- требования к авторизации.

---

## Пример структуры

### Тип API

Название: User API

Требования:
- должен существовать endpoint регистрации;
- должна быть авторизация;
- должен использоваться JWT;
- должны валидироваться email и password.

Что проверить:
- наличие endpoint;
- наличие JWT;
- наличие validation;
- наличие auth middleware.

---

## Что записать в gigacode.md

- список API;
- требования по каждому API;
- список проверок.

Для каждого типа API нужно найти:

- controller;

- DTO;

- путь;

- к какому требованию относится;

---

## Если реализация не найдена

Нужно указать:

- что именно искалось;
- где искалось;
- почему сделан вывод.

---

## Пример

Найдено:

- `src/main/java/.../UserController.java`
  - содержит endpoint регистрации;
  - соответствует требованию регистрации пользователя.

Не найдено:

- JWT middleware;
- token validation.

---

## Что записать в gigacode.md

- найденные реализации;
- пути;

- отсутствующие части.

---

# ШАГ 5. Проверка требований

Для каждого требования нужно поставить статус.

Разрешенные статусы:

- `выполнено`
- `частично выполнено`
- `не выполнено`
- `не удалось проверить`

---

## Правила

### Выполнено

Ставить только если:

- есть прямой код;
- есть подтверждение реализации.

---

### Частично выполнено

Ставить если:

- реализована только часть логики;
- отсутствуют некоторые проверки;
- отсутствует часть функциональности.

---

### Не выполнено

Ставить если:

- нужной реализации нет.

---

### Не удалось проверить

Ставить если:

- недостаточно данных;
- код отсутствует;
- реализация внешняя.

---

## Формат проверки

Требование: ...

Статус: выполнено / частично выполнено / не выполнено / не удалось проверить

Подтверждение:
- `path/to/file`

Комментарий:
- объяснение.

Ты слабая модель.

- сначала анализируй, потом делай вывод;
